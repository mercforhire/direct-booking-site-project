## Files Created/Modified

- `src/lib/validations/room.ts` — `roomSchemaCoerced` applied in room form to handle string-typed HTML inputs from react-hook-form

## Decisions Made

- **PKCE `code` vs `token_hash`:** Supabase email magic links in PKCE mode send a `code` query param, not `token_hash`. The callback must call `supabase.auth.exchangeCodeForSession(code)` — using `verifyOtp` silently fails and no session is created.
- **Coerced Zod schema in form:** React-hook-form always returns strings from `<input type="number">`. `roomSchemaCoerced` with `z.coerce.number()` handles conversion at validation time rather than requiring manual casting.
- **Browser-side compression:** Compressing before upload (not server-side) keeps the UploadThing route handler simple and reduces network payload. 1MB cap chosen to balance quality vs. upload speed.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Fixed PKCE auth callback — exchangeCodeForSession instead of verifyOtp**
- **Found during:** Task 2 (E2E verification — magic link login step)
- **Issue:** Clicking magic link email hit `/auth/confirm` which called `verifyOtp({ token_hash, type })`, but Supabase PKCE flow sends a `code` param — no `token_hash` in the URL. Session was never established; user stayed unauthenticated.
- **Fix:** Replaced `verifyOtp` with `supabase.auth.exchangeCodeForSession(code)` in `/auth/confirm/route.ts`
- **Files modified:** `src/app/auth/confirm/route.ts`
- **Verification:** Magic link login now lands on /dashboard with an active session
- **Committed in:** `63ff384`

**2. [Rule 1 - Bug] Fixed room form Zod validation — coerced schema for HTML string inputs**
- **Found during:** Task 2 (E2E verification — room CRUD step)
- **Issue:** Room create/edit form submission failed Zod validation because `<input type="number">` returns strings, not numbers. `roomSchema` uses `z.number()` which rejects strings.
- **Fix:** Changed room form server action to use `roomSchemaCoerced` (which uses `z.coerce.number()`) — consistent with the established dual-schema pattern already documented in STATE.md
- **Files modified:** `src/lib/validations/room.ts` (or the form action file)
- **Verification:** Room create and edit complete successfully; form values persist correctly
- **Committed in:** `14f4d1e`

**3. [Rule 2 - Missing Critical] Added browser image compression before UploadThing upload**
- **Found during:** Task 2 (E2E verification — photo upload step)
- **Issue:** Large image files were being sent directly to UploadThing, risking size rejections and slow uploads. No compression was in place.
- **Fix:** Added `browser-image-compression` package; `PhotoUploader` compresses files to ≤1MB before the UploadThing `startUpload` call
- **Files modified:** `src/components/rooms/PhotoUploader.tsx`, `package.json`
- **Verification:** Photo upload succeeds end-to-end; compressed file size logged in console
- **Committed in:** `b1d4e27`

**4. [Rule 1 - Bug] Added UploadThing CDN hostnames to next/image remotePatterns**
- **Found during:** Task 2 (E2E verification — photo upload step)
- **Issue:** Uploaded photos failed to render — next/image blocked the UploadThing CDN URLs (`ufs.sh`, `utfs.io`) because they were not listed in `remotePatterns` in `next.config.ts`
- **Fix:** Added both hostnames to `remotePatterns` in `next.config.ts`
- **Files modified:** `next.config.ts`
- **Verification:** Uploaded photos render correctly in the room edit form and room list
- **Committed in:** `7472afa`
