<task type="auto">
  <name>Task 1: Update Prisma schema — remove NextAuth models and add directUrl</name>
  <files>prisma/schema.prisma</files>
  <action>
    Rewrite prisma/schema.prisma with these two changes:

    1. Update the datasource block to add directUrl:
    ```prisma
    datasource db {
      provider  = "postgresql"
      url       = env("DATABASE_URL")
      directUrl = env("DIRECT_URL")
    }
    ```

    2. Delete the four NextAuth models entirely: Account, Session, User, VerificationToken. Remove the comment "// Auth.js required models" as well.

    Keep the generator block and all four application models unchanged:
    - model Room (with all its fields, photos and addOns relations)
    - model RoomPhoto (with all its fields and room relation)
    - model AddOn (with all its fields and room relation)
    - model Settings (with id defaulting to "global")

    Do NOT add a User/Profile model — Supabase manages user identity in its own auth schema which Prisma does not touch.

    After editing, run `npx prisma generate` to regenerate the Prisma client. This removes the Account/Session/User/VerificationToken types from the client — expect TypeScript errors in files that import those types (they will be fixed in Plan 04).
  </action>
  <verify>
    <automated>npx prisma validate</automated>
  </verify>
  <done>prisma validate passes. Schema has directUrl in datasource. No Account, Session, User, or VerificationToken models present. Room, RoomPhoto, AddOn, Settings models unchanged.</done>
</task>

<task type="auto">
  <name>Task 2: Push schema to Supabase PostgreSQL</name>
  <files>prisma/schema.prisma</files>
  <action>
    Push the Prisma schema to Supabase using the direct URL (required because PgBouncer transaction mode doesn't support DDL statements):

    ```bash
    npx prisma db push
    ```

    Prisma will use DIRECT_URL for this operation automatically because it is configured in the schema.

    If the push fails with a connection error, verify:
    - DATABASE_URL and DIRECT_URL are set in .env.local
    - DATABASE_URL uses port 6543 with ?pgbouncer=true appended
    - DIRECT_URL uses port 5432 with no extra params
    - The Supabase project is fully provisioned (not still initializing)

    After a successful push, verify the tables exist by running:
    ```bash
    npx prisma db execute --stdin <<< "SELECT table_name FROM information_schema.tables WHERE table_schema = 'public';"
    ```

    Expected output includes: Room, RoomPhoto, AddOn, Settings (Prisma uses quoted identifiers — table names match model names).
  </action>
  <verify>
    <automated>npx prisma db execute --stdin <<< "SELECT COUNT(*) FROM information_schema.tables WHERE table_schema = 'public' AND table_name IN ('Room', 'RoomPhoto', 'AddOn', 'Settings');"</automated>
  </verify>
  <done>prisma db push succeeds without error. All four application tables (Room, RoomPhoto, AddOn, Settings) exist in Supabase PostgreSQL public schema. No NextAuth tables (Account, Session, User, VerificationToken) exist in the public schema.</done>
