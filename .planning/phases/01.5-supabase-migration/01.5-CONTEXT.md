# Phase 1.5: Supabase Migration - Context

**Gathered:** 2026-03-26
**Status:** Ready for planning
**Source:** User instructions

<domain>
## Phase Boundary

Replace the existing local PostgreSQL + NextAuth v5 setup with Supabase managed PostgreSQL and Supabase Auth. Prisma remains as the ORM. UploadThing remains for file storage. All existing admin features (room CRUD, photo management, settings) must continue to work after migration.

</domain>

<decisions>
## Implementation Decisions

### Database
- Use Supabase managed PostgreSQL as the database backend
- Keep Prisma as ORM (no switch to Supabase client for data queries)
- Point DATABASE_URL to Supabase connection string
- Run prisma db push to migrate schema to Supabase

### Auth
- Replace next-auth v5 + @auth/prisma-adapter with Supabase Auth
- Remove packages: next-auth, @auth/prisma-adapter
- Add packages: @supabase/supabase-js, @supabase/ssr
- Auth method: magic link / OTP email (same UX as current Resend-based flow)
- Supabase Auth manages its own auth tables in the auth schema — remove Prisma models: Account, Session, VerificationToken, and auth fields from User

### Prisma Schema
- Remove models: Account, Session, VerificationToken
- Remove or simplify User model (Supabase Auth manages user identity)
- Keep models: Room, RoomPhoto, AddOn, Settings
- If a User/Profile model is needed to link to Supabase auth.users, use UUID foreign key (auth.users.id)

### Supabase Client Utilities
- Create browser client: src/lib/supabase/client.ts
- Create server client: src/lib/supabase/server.ts (using @supabase/ssr createServerClient with cookies)
- Create middleware client: used in src/middleware.ts

### Middleware
- Replace NextAuth middleware with Supabase SSR middleware
- Protect routes: /dashboard, /rooms, /settings
- Redirect unauthenticated users to /login

### Server Actions Auth
- Replace all auth() session checks (from next-auth) with Supabase createServerClient session checks
- All server actions: room.ts, room-photos.ts, settings.ts

### Login Page
- Replace NextAuth signIn() with Supabase magic link / OTP email flow
- User enters email → Supabase sends magic link → user clicks link → authenticated

### Environment Variables
- Add: NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY
- Remove: NEXTAUTH_SECRET, NEXTAUTH_URL, AUTH_RESEND_KEY, AUTH_FROM_EMAIL
- Keep: DATABASE_URL (pointing to Supabase), RESEND_API_KEY (may use for custom SMTP or remove if Supabase handles email), UPLOADTHING_SECRET, UPLOADTHING_APP_ID

### File Storage
- Keep UploadThing for photo uploads (no migration to Supabase Storage)

### Claude's Discretion
- Whether to configure custom SMTP in Supabase (using Resend) or use Supabase's built-in email
- Exact Supabase project region selection
- Error handling on auth callbacks
- Auth callback route implementation (if needed for magic link redirect)

</decisions>

<specifics>
## Specific Ideas

- Use @supabase/ssr (not the legacy auth-helpers) — this is the current recommended approach for Next.js App Router
- Supabase magic link flow: user submits email → call supabase.auth.signInWithOtp({ email }) → Supabase sends email → user clicks link → handled by /auth/callback route → session set via cookies
- Server client must use next/headers cookies() for reading/writing session cookies
- Middleware must refresh session on every request using updateSession pattern

</specifics>

<deferred>
## Deferred Ideas

- Supabase Storage for photo uploads (keeping UploadThing)
- Row Level Security (RLS) policies (admin-only app, not needed now)
- Supabase Realtime features

</deferred>
