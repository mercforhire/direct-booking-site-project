## Summary

This phase replaces two systems: the NextAuth v5 + Prisma adapter auth stack (which manages its own session tables), and the local PostgreSQL instance, both replaced by Supabase. Supabase provides managed PostgreSQL (accessed via Prisma) and Supabase Auth (replacing NextAuth's session/account models). Prisma stays as the ORM for application data (Room, RoomPhoto, AddOn, Settings); Supabase Auth handles identity internally in its own `auth` schema, which Prisma does not touch.

The migration has four mechanical steps: (1) update the Prisma schema to remove NextAuth models and add dual connection URLs, (2) install Supabase packages and create three client utilities, (3) rewrite middleware.ts using the Supabase SSR `updateSession` pattern, and (4) rewrite all server actions and the login page to use Supabase session checks instead of NextAuth's `auth()`.

One important nuance: in Next.js 15, `cookies()` from `next/headers` is async and must be awaited before passing to `createServerClient`. Community sources show this is the established workaround for the cookie-handling compatibility gap between `@supabase/ssr` and Next.js 15. The auth callback route (`/auth/confirm`) is needed to exchange the magic link `token_hash` for a session when the user clicks the email link.

**Primary recommendation:** Use `NEXT_PUBLIC_SUPABASE_ANON_KEY` (not the newer `PUBLISHABLE_KEY` naming) since the project's existing library versions predate the key renaming rollout; both work during the transition period. Configure Resend as custom SMTP in Supabase dashboard to retain deliverability and domain branding.

### Pattern 1: Browser Client (Client Components)

**What:** Singleton browser client for use in Client Components.
**When to use:** Any `"use client"` file that needs Supabase auth (e.g., login page calling `signInWithOtp`).

```typescript
// src/lib/supabase/client.ts
import { createBrowserClient } from "@supabase/ssr"

export const createClient = () =>
  createBrowserClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  )
```

Source: Verified via official Supabase SSR docs + community examples (MEDIUM confidence — pattern consistent across multiple sources).

### Pattern 2: Server Client (Server Components, Server Actions, Route Handlers)

**What:** Async server client that awaits cookies() for Next.js 15 compatibility.
**When to use:** Any server-side code — Server Components, Server Actions, Route Handlers.

**Critical Next.js 15 note:** `cookies()` from `next/headers` is async in Next.js 15. The server client factory must be `async` and must `await cookies()` before passing to `createServerClient`. This is a breaking change from Next.js 14 patterns.

```typescript
// src/lib/supabase/server.ts
import { createServerClient } from "@supabase/ssr"
import { cookies } from "next/headers"

export async function createClient() {
  const cookieStore = await cookies()

  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return cookieStore.getAll()
        },
        setAll(cookiesToSet) {
          try {
            cookiesToSet.forEach(({ name, value, options }) =>
              cookieStore.set(name, value, options)
            )
          } catch {
            // Server Components cannot write cookies — middleware handles persistence
          }
        },
      },
    }
  )
}
```

Source: ryankatayi.com (verified against Supabase discussion #35981 re: async cookies) — MEDIUM confidence.

### Pattern 3: Middleware — updateSession

**What:** Middleware refreshes the Supabase session on every request and syncs auth cookies between request and response. Route protection logic lives here.
**When to use:** Every request — middleware runs before all route handlers.

**Critical note:** Use `supabase.auth.getUser()` in middleware, NOT `getSession()`. `getUser()` validates the JWT with the Supabase Auth server on every call. `getSession()` only reads local state and cannot be trusted for protection logic (official Supabase guidance).

```typescript
// src/middleware.ts
import { createServerClient } from "@supabase/ssr"
import { NextResponse, type NextRequest } from "next/server"

export async function middleware(request: NextRequest) {
  let supabaseResponse = NextResponse.next({ request })

  const supabase = createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() {
          return request.cookies.getAll()
        },
        setAll(cookiesToSet) {
          // Sync cookies to both the request (for Server Components) and response (for browser)
          cookiesToSet.forEach(({ name, value }) =>
            request.cookies.set(name, value)
          )
          supabaseResponse = NextResponse.next({ request })
          cookiesToSet.forEach(({ name, value, options }) =>
            supabaseResponse.cookies.set(name, value, options)
          )
        },
      },
    }
  )

  // IMPORTANT: getUser() not getSession() — validates token with auth server
  const { data: { user } } = await supabase.auth.getUser()

  const adminPaths = ["/dashboard", "/rooms", "/settings"]
  const isAdminRoute = adminPaths.some((p) =>
    request.nextUrl.pathname.startsWith(p)
  )

  if (!user && isAdminRoute) {
    const url = request.nextUrl.clone()
    url.pathname = "/login"
    return NextResponse.redirect(url)
  }

  return supabaseResponse
}

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico).*)"],
}
```

Source: Synthesized from official pattern + community verified examples — MEDIUM confidence. The `setAll` pattern (iterating cookies manually rather than calling `setAll` on response) is the correct approach because `response.cookies.setAll()` does not exist on Next.js `ResponseCookies`.

### Pattern 4: Auth Callback Route (Magic Link Exchange)

**What:** Route handler at `/auth/confirm` that exchanges the `token_hash` from the magic link URL for a session cookie.
**When to use:** Required — Supabase magic links redirect to this route after the user clicks the link in email. Without this, the session is never established.

**How magic link flow works:**
1. User submits email on login page → `supabase.auth.signInWithOtp({ email, options: { emailRedirectTo: 'http://localhost:3000/auth/confirm' } })`
2. Supabase sends email with link containing `?token_hash=...&type=email`
3. User clicks link → browser goes to `/auth/confirm?token_hash=...&type=email`
4. Route handler calls `supabase.auth.verifyOtp({ type, token_hash })` → session established in cookies
5. Redirect to `/dashboard`

```typescript
// src/app/auth/confirm/route.ts
import { type EmailOtpType } from "@supabase/supabase-js"
import { type NextRequest, NextResponse } from "next/server"
import { createClient } from "@/lib/supabase/server"
import { redirect } from "next/navigation"

export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url)
  const token_hash = searchParams.get("token_hash")
  const type = searchParams.get("type") as EmailOtpType | null
  const next = searchParams.get("next") ?? "/dashboard"

  if (token_hash && type) {
    const supabase = await createClient()
    const { error } = await supabase.auth.verifyOtp({ type, token_hash })
    if (!error) {
      redirect(next)
    }
  }

  // Token invalid or missing — redirect to login with error
  redirect("/login?error=invalid_token")
}
```

Source: ryankatayi.com + Supabase magic link docs — MEDIUM confidence.

### Pattern 5: Auth Guard in Server Actions

**What:** Replace `auth()` from NextAuth with `createClient()` + `getUser()` from Supabase.
**When to use:** All server actions that require authentication.

```typescript
// Before (NextAuth):
async function requireAuth() {
  const session = await auth()
  if (!session?.user) throw new Error("Unauthorized")
  return session
}

// After (Supabase):
async function requireAuth() {
  const supabase = await createClient()
  const { data: { user }, error } = await supabase.auth.getUser()
  if (error || !user) throw new Error("Unauthorized")
  return user
}
```

### Pattern 6: Login Page (Client Component)

**What:** Replace `signIn("resend", ...)` with `supabase.auth.signInWithOtp({ email })`.

```typescript
// "use client"
import { createClient } from "@/lib/supabase/client"

async function handleSubmit(e: React.FormEvent) {
  e.preventDefault()
  const supabase = createClient()
  await supabase.auth.signInWithOtp({
    email,
    options: {
      shouldCreateUser: false, // admin only — don't auto-create new users
      emailRedirectTo: `${window.location.origin}/auth/confirm`,
    },
  })
  setSent(true)
}
```

**Critical:** `shouldCreateUser: false` prevents Supabase from auto-registering any email that submits the form. The landlord account must be created manually in the Supabase dashboard first.

### Anti-Patterns to Avoid

- **Using `getSession()` for protection:** `getSession()` reads local cookie state without revalidating with the auth server. Always use `getUser()` in server-side protection logic.
- **Not awaiting `cookies()` in Next.js 15:** The server client factory must be `async` and `await cookies()` — omitting this causes TypeScript errors and runtime failures.
- **Using legacy `@supabase/auth-helpers-nextjs`:** This package is deprecated. Only use `@supabase/ssr`.
- **Calling `response.cookies.setAll()` in middleware:** This method does not exist on Next.js `ResponseCookies`. Iterate with `forEach` instead.
- **Forgetting `emailRedirectTo` in `signInWithOtp`:** Without this, Supabase uses the Site URL from project settings. In development, it must point to `http://localhost:3000/auth/confirm`. Configure allowed redirect URLs in Supabase Auth dashboard.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session cookie management | Custom JWT cookie read/write | `@supabase/ssr` createServerClient cookie adapter | PKCE flow, token refresh, secure cookie options handled automatically |
| Token refresh on expiry | Manual refresh logic in middleware | `supabase.auth.getUser()` in middleware (triggers refresh internally) | Supabase handles token rotation transparently |
| Magic link token exchange | Custom URL parsing + DB token lookup | `supabase.auth.verifyOtp({ type, token_hash })` | Supabase validates PKCE code verifier + token hash |
| User existence check | Query Prisma User table in actions | `supabase.auth.getUser()` | Auth is Supabase's domain; User table no longer exists in app schema |
| Password reset / account recovery | Custom email + token flow | Supabase Auth built-in flows | Complex security-sensitive logic already implemented |

## Common Pitfalls

### Pitfall 1: `getSession()` vs `getUser()` confusion

**What goes wrong:** Developer uses `supabase.auth.getSession()` to protect server routes and actions. The session object is read from cookie without network validation, so an expired or tampered token passes the check.
**Why it happens:** `getSession()` is the intuitive name for "get the current session" and works fine client-side where freshness is enforced by the browser. Server-side it reads stale data.
**How to avoid:** Use `supabase.auth.getUser()` in all server-side code. It sends a request to Supabase Auth to revalidate the JWT every time.
**Warning signs:** Session appears valid but API calls fail; expired sessions don't trigger redirects to login.

### Pitfall 2: `cookies()` not awaited in Next.js 15

**What goes wrong:** TypeScript error "Property 'get' does not exist on type 'Promise'" when constructing server client. Or runtime error accessing cookie methods.
**Why it happens:** Next.js 15 made `cookies()` from `next/headers` return a Promise. Old examples and older Supabase docs show synchronous `cookies()`.
**How to avoid:** Make `createClient()` in `server.ts` an `async` function and `const cookieStore = await cookies()` before using it.
**Warning signs:** TypeScript complaining about Promise type; cookies not being set after login.

### Pitfall 3: Magic link redirects to wrong URL

**What goes wrong:** User clicks magic link and lands on Supabase's Site URL instead of the app's `/auth/confirm` route. Session is never established.
**Why it happens:** `emailRedirectTo` not passed to `signInWithOtp`, or the URL is not in Supabase's allowed redirect list.
**How to avoid:** (a) Pass `emailRedirectTo` explicitly in every `signInWithOtp` call. (b) Add `http://localhost:3000/**` and production URL to Supabase Auth > URL Configuration > Redirect URLs.
**Warning signs:** Magic link works on Supabase side but user ends up on a 404 or Supabase default page.

### Pitfall 4: `shouldCreateUser` not set to `false`

**What goes wrong:** Any email submitted on the login form creates a new Supabase Auth user. Strangers can create accounts.
**Why it happens:** Supabase default for `signInWithOtp` is `shouldCreateUser: true`.
**How to avoid:** Always pass `shouldCreateUser: false` in the login page's `signInWithOtp` call. Pre-create the landlord account manually in Supabase Dashboard > Authentication > Users.
**Warning signs:** Unknown emails receiving magic links; unexpected users appearing in auth.users.

### Pitfall 5: Middleware cookie sync pattern broken

**What goes wrong:** Auth cookie is not propagated to Server Components after middleware refresh, or session disappears after first request.
**Why it happens:** The `setAll` in middleware must update BOTH `request.cookies` (for Server Components to read in the same request) AND `supabaseResponse.cookies` (for the browser to receive). If only one is updated, the chain breaks.
**How to avoid:** In middleware `setAll`, rebuild `supabaseResponse = NextResponse.next({ request })` after updating `request.cookies`, then set cookies on the new response.
**Warning signs:** User authenticated in browser but server components see no session; random logouts between requests.

### Pitfall 6: Prisma schema still references removed models

**What goes wrong:** `prisma db push` fails or app crashes because code still imports `prisma.user`, `prisma.session`, etc.
**Why it happens:** The Account, Session, User, VerificationToken models are removed from schema, but imports in server actions or auth files still reference them.
**How to avoid:** Grep codebase for `prisma.user`, `prisma.session`, `prisma.account`, `prisma.verificationToken` before running db push. Remove all references.
**Warning signs:** TypeScript errors on `prisma.` method calls; Prisma generate errors.

### Pitfall 7: `RESEND_API_KEY` vs Supabase SMTP

**What goes wrong:** Emails not delivered because `AUTH_RESEND_KEY` was removed but Supabase's built-in SMTP is rate-limited or sends from Supabase domain.
**Why it happens:** Supabase built-in SMTP has unspecified rate limits and sends from a Supabase-owned domain. Without custom SMTP configuration, magic link emails may fail in production.
**How to avoid:** Configure Resend as custom SMTP in Supabase Dashboard > Authentication > SMTP Settings. Use the existing `RESEND_API_KEY` value; Resend provides SMTP credentials (host: `smtp.resend.com`, port: `465`, username: `resend`, password: `RESEND_API_KEY`).

## Open Questions

1. **Exact `@supabase/ssr` version supporting Next.js 15**
   - What we know: Community reports of cookie issues with some versions; pattern of awaiting `cookies()` resolves the TypeScript issue; at least one production template (SupabaseAuthWithSSR) works with Next.js 15
   - What's unclear: Minimum version of `@supabase/ssr` where Next.js 15 works reliably; whether the `await cookies()` pattern is needed in all versions
   - Recommendation: Install latest `@supabase/ssr`, implement the `await cookies()` async pattern from the start, test immediately after setup

2. **Supabase SMTP rate limits in development**
   - What we know: Built-in SMTP has "significant rate limits" per Supabase docs; custom SMTP (Resend) recommended for production
   - What's unclear: Whether built-in SMTP is sufficient for development/staging (just testing magic links with one email)
   - Recommendation: Configure Resend SMTP from the start to avoid switching mid-development; Supabase + Resend integration is documented and straightforward

3. **Supabase project region**
   - What we know: Marked as Claude's discretion in CONTEXT.md
   - What's unclear: No project location specified
   - Recommendation: Choose `us-east-1` (AWS US East) or `ca-central-1` (AWS Canada) depending on intended user geography; choose closest to deployment region
