## Critical Issues (Blockers)

### P0 — Middleware Regression: Token-Only Guests Blocked (BOOK-05, GUEST-01)

**File:** `src/middleware.ts`
**Introduced:** Phase 5 commit `bd30fcf`

`adminPaths` array includes `"/bookings"` to protect the admin booking list at `/bookings`. However the guest detail route `/bookings/[id]` shares this prefix. Guests who submitted a booking without creating an account (BOOK-05) are redirected to the admin OTP login page when following their access link or returning from Stripe. The token at `?token=xxx` never reaches the page code.

**Affected flows:**
- Guest submits booking without account → redirect to `/bookings/[id]?token=xxx&new=1` → middleware intercepts → redirected to `/login`
- Booking confirmation email link → same fate
- Approval email link → same fate
- Any email CTA for a no-account guest → broken

**Fix:** Use an exact-match or more specific prefix for the admin booking list (`/admin/bookings` is already protected; `/bookings` page.tsx appears to be at `src/app/(admin)/bookings/page.tsx` — check whether it should be moved under `/admin/`).

### P1 — Date Change Top-Up: Missing Page Fallback + No Confirmation Email

**File:** `src/actions/date-change.ts` (lines 186, 390), `src/app/bookings/[id]/page.tsx`

`approveDateChange` generates a Stripe session with `success_url: .../bookings/${id}?date_change_paid=1`. The booking page's `searchParams` type does not include `date_change_paid`, and there is no page-load webhook fallback for this payment type. Additionally, the webhook handler for `date_change_topup` sends no confirmation email (unlike booking and extension payments).
